Foundation / Design Tokens

Design Tokens Documentation

This documentation translates the current token exports into a structured design system reference. It defines how foundational values should be organised, named, and used across product design and implementation.

Overview

The token system is organised into four layers:

Primitives

Raw, reusable values — color scales, spacing values, and base constants.

Semantic Tokens

Role-based aliases — text-primary, bg-primary, spacing-md.

Modes

Light and dark theme mappings — same semantic token resolves differently by context.

Value Tokens

Typography and other foundational values used to build styles.

Token Architecture
1. Primitive Tokens

Primitive tokens are the source of truth for raw values. They should not describe intent or component usage. Instead, they define the underlying scales used by the system.

Examples:

Colors/Base/white Colors/Gray (light mode)/900 Colors/NCB Blue/500 Spacing/4 (16px)

Primitive tokens should be: stable · reusable · mode-agnostic · named by family and step, not by UI meaning.

2. Semantic Tokens

Semantic tokens map primitives to interface roles. They are the tokens designers and developers should use in product work.

text-primary text-secondary bg-primary bg-disabled spacing-md

Semantic tokens should: communicate purpose · remain consistent even when primitive values change · support mode switching without changing component logic.

3. Modes

Modes allow the same semantic token to resolve differently in light and dark contexts.

text-primary → #111111 (dark neutral)
text-primary → #F5F5F5 (light neutral)

This preserves role consistency while adapting to surface contrast.

4. Value Tokens

Value tokens define non-color foundations such as typography families, font weights, sizes, and line heights. These values support text styles and should be referenced by higher-order style tokens rather than applied ad hoc.

Color Foundations
Base Colors

Base colors establish universal constants used across the system.

white

#FFFFFF

black

#000000

transparent

rgba(0,0,0,0)

Use base tokens only when a universal constant is needed. In most product work, semantic tokens should be preferred.

Neutral Color System

The system includes multiple neutral families, each designed for different visual needs.

Core Neutral Families
Gray (light mode) Gray (dark mode) Gray (dark mode alpha)
Scale Pattern — 25 to 950

25–100 — lightest surfaces, subtle fills, low emphasis backgrounds

200–400 — borders, disabled states, secondary surfaces

500–700 — medium emphasis text, icons, controls

800–950 — strongest contrast, headings, foreground on light surfaces

Brand Colors
NCB Blue — Primary Brand

Recommended roles: primary actions · interactive states · links · selected states · branded surfaces.

NCB Yellow — Secondary Brand

Recommended roles: highlights · callouts · badges · brand accents · celebratory or high-visibility moments.

Blue should remain the primary functional brand color. Yellow should be used more selectively and should not carry large volumes of text or UI chrome without contrast validation.

Feedback Colors

Error

Destructive actions, validation failures, critical alerts.

Warning

Cautionary messaging, pending issues, elevated attention states.

Success

Confirmations, completed actions, positive outcomes.

Use lighter steps for background fills and status badges, medium steps for borders and icons, and darker steps for high-contrast text or emphasis.

Background Tokens

Background tokens define primary surfaces, alternates, states, and layered fills.

TokenUsage
bg-primaryMain page/surface background
bg-primary_altAlternate surface, subtle distinction
bg-primary_hoverHover state for primary surfaces
bg-quaternaryDeeply nested or low-emphasis fills
bg-activeActive/selected state background
bg-disabledDisabled element background

Use background tokens to preserve consistent elevation and contrast across themes. Hover, active, and disabled background states should always be driven by semantic tokens.

Naming Conventions
Primitive Naming

Pattern: Category / Family / Step

Colors / NCB Blue / 500 Colors / Gray (light mode) / 900 Spacing / 4 (16px)
Semantic Naming

Pattern: role-state-variant

text-primary text-secondary_hover bg-disabled spacing-md
Naming Principles
  • Name by purpose, not appearance, at the semantic layer
  • Name by family and scale, not purpose, at the primitive layer
  • Avoid duplicate tokens that express the same UI meaning
  • Avoid using raw hex values or pixel values directly in product components
Usage Rules
Designers
  • Apply semantic tokens in components and templates
  • Use primitives only when defining or revising foundations
  • Do not manually substitute similar gray values when a semantic token already exists
  • Validate yellow and low-contrast color pairings carefully
Developers
  • Consume semantic tokens in UI code
  • Keep primitive references inside the token pipeline rather than application code
  • Support theme switching through mode-aware token resolution
  • Do not hardcode color or spacing values already represented by tokens
Accessibility

Semantic tokens should be evaluated in context, especially for:

text on brand surfaces disabled content subtle borders warning and success backgrounds dark theme overlays and alpha values
Governance
Add a new primitive token when:
  • No existing family and scale step can support the need
  • The value has clear reuse potential
  • The value represents a foundational addition, not a one-off component exception
Add a new semantic token when:
  • A new UI role is needed
  • An existing role is overloaded or ambiguous
  • A token must behave differently across themes while preserving intent
Avoid:
  • Component-specific primitive names
  • Ad hoc one-off colors
  • Inconsistent spacing aliases
  • Publishing internal or experimental palettes without a defined use case

Summary

This token system is already structured strongly enough to support a mature design system. The main documentation goal is to separate what values exist from how they should be used.

Primitives define the scales

Semantic tokens define intent

Modes define context

Value tokens define foundational typographic behaviour
